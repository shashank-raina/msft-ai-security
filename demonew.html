Entra Agent ID — Concept
What it is, why it matters, how Classic vs Modern agents differ
🚨 The problem today: Most Copilot Studio agents in production are "Classic" — registered as ordinary service principals with zero agent-specific security controls. No CA for Agents, no ID Protection, no lifecycle governance. Entra Agent ID fixes this — but it is Preview, Frontier programme only, and only covers agents created after enabling it.
💡 Plain English: Today most agents share the builder's identity badge. Entra Agent ID gives each agent its own badge — with its own permissions, risk detection, and owner accountability. The Security Dashboard only gets rich data for agents with their own badge (Modern agents).
Classic vs Modern — the split that matters most
⚠ Classic Agents (majority)
Registered as ordinary service principals. No CA for Agents, no ID Protection, no lifecycle governance. Use OBO, maker credentials, or App Registration patterns. Most existing Copilot Studio production agents are Classic. Migration tool does not exist yet.
✓ Modern Agents (Entra Agent ID)
Purpose-scoped identity per agent. Full security stack: CA for Agents, ID Protection, lifecycle governance (Owner + Sponsor). Requires Frontier programme. New agents only — existing Classic agents need manual migration.

The four identity objects
📋
Agent Identity Blueprint
Template + Credential Container
Defines how agents are created. Holds credentials (FIC recommended — no stored secret). One Blueprint can create many Agent Identities. Blueprint count is a security boundary decision — a single credential compromise affects all child agents. If deleted: all child agents lose auth but permissions remain.
⚠ Protect credentials here
🔑
Blueprint Principal
Audit Identity
Service principal created when blueprint is added to tenant. One job only: create/deprovision agent identities. All lifecycle events (creation, deletion) are logged under this identity in Entra audit logs. If deleted: the audit trail for all agents it created is severed.
⚠ Deletion breaks audit trail
🤖
Agent Identity
The Agent's Service Principal
No credentials of its own — Blueprint performs token exchange via impersonation model. Blueprint appears in logs during T1; Agent Identity appears as client in T2 access tokens and all downstream audit logs.

3 security guarantees:
① No admin token generation — not even Global Admins can generate agent identity tokens. Microsoft controls the Blueprint.
② Tenant-bound — tokens only valid in home tenant.
③ Impersonation model — Blueprint compromise scope is bounded by Blueprint count.
✓ Full Entra security stack applies
👤
Agent User
Optional — Full Human Identity
Some agents get a full Entra user account — mailbox, Teams presence, SharePoint access, MFA settings. Appears as a completely normal user with no visual indicator it belongs to an agent. Very high risk: if Blueprint is deleted, Agent User remains active with all permissions and no flag indefinitely.
🚨 Survives Blueprint deletion silently

Agent segmentation — custom security attributes
The recommended approach for CA policy scoping at scale. Instead of managing object IDs, assign custom security attributes to agents and target them in CA policies.

Example pattern: Create attribute set AgentAttributes with attribute AgentApprovalStatus (values: New, In_Review, HR_Approved, Finance_Approved, IT_Approved). Tag agents with their approval status. CA policy: block all agents except those tagged HR_Approved from accessing HR resources. Scales to hundreds of agents without per-object configuration.

Microsoft Agent 365
The enterprise control plane for AI agents — GA May 1, 2026
✅ GA May 1, 2026 $15/user/month standalone Included in M365 E7 ($99/user/mo)
💡 What Agent 365 actually is: Not an agent builder. Not a hosting platform. It is a management and security layer that wraps agents you have already built — regardless of which platform they run on. Copilot Studio, Foundry, LangChain, OpenAI SDK, AWS Bedrock, GCP Vertex AI — once an agent is onboarded, it gets enterprise controls it did not have before.
Four things Agent 365 adds to any agent
🪪
Entra Identity
Every onboarded agent gets an Entra Agent ID. Enables CA policies, ID Protection risk detection, lifecycle governance (Owner + Sponsor). SDK handles Entra registration automatically.
🛡️
ATG Real-Time Protection
Agent Tooling Gateway — evaluates every tool invocation before execution. Blocks unsafe actions in real time via Defender. Critical limitation: ATG only inspects the tool execution path, not model reasoning between tool calls.
📊
Observability
OpenTelemetry instrumentation via Agent 365 SDK. All inference events, tool usage, and interactions routed to M365 audit logs. Visible in Defender Advanced Hunting via AIAgentsInfo table with RegistrySource = "A365".
📋
Governance Registry
All agents appear in M365 admin center → Agents → All agents. Roles: AI Administrator, AI Reader. No licence needed to view inventory — just the role. RegistrySource = "A365" in AIAgentsInfo.
Platform support — fully platform-agnostic
🤖 Copilot Studio 🏭 Microsoft Foundry 🔗 LangChain SDK 🤖 OpenAI Agents SDK 🦾 Claude Code SDK ⚙️ Microsoft Agent Framework ☁️ AWS Bedrock 🌐 GCP Vertex AI
You add the Agent 365 SDK to your agent code. The SDK handles Entra identity registration, OpenTelemetry event emission, and ATG integration automatically. You do not rewrite your agent — you add a governance layer around it. Works on Azure, AWS, GCP, or your own infrastructure.

ATG — the critical limitation
⚠️ ATG only operates on the tool execution path. It evaluates tool invocations (what the agent tries to DO) but does NOT inspect model reasoning between tool calls. An agent can reason about malicious instructions, decide to act on them, and only get caught at the moment it tries to execute a tool — if Defender responds within 1 second. Attacker-crafted inputs targeting the reasoning layer (prompt injection, XPIA) are not directly blocked by ATG.

Setting Up Copilot Monitoring
Dual-admin setup — Defender admin and Power Platform admin must work together
⚠️
Two admins required — neither can complete this aloneThe Power Platform admin needs a URL that only the Defender admin can generate. Coordinate before starting.
1
Enable Security for AI agents in Defender
🔵 Defender Admin Defender portal → Settings → Security for AI agents → Enable toggle
Turns on the Security for AI feature and creates the integration point Copilot Studio registers with. Previously at Settings → Cloud Apps → AI Agents — navigation moved April 2026.
2
Connect Microsoft 365
🔵 Defender Admin Settings → Security for AI agents → Connect Microsoft 365
Enables Copilot agent telemetry to flow into Defender. Without this you get inventory but no alerts or behavioural data.
3
Copy the webhook URL
🔵 Defender Admin Security for AI agents → Copilot Studio AI Agents → Copy URL
Defender generates a unique tenant-specific URL. Pass this to the Power Platform admin — they need it for Step 4.
4
Enable Defender in Power Platform
🟣 Power Platform Admin admin.powerplatform.com → Security → Threat Detection → Microsoft Defender → Enable
Paste the webhook URL from Step 3. This tells Copilot Studio to send agent telemetry to Defender.
⚠️ Requires Power Platform Admin role — Defender Admin cannot do this step.
5
Enable Real-Time Protection
🔵 Defender Admin Settings → Security for AI agents → Real-time protection → Enable
Enables Defender to evaluate and block tool invocations before execution.
⚠️ 1-second timeout — fast tool calls may execute before a block decision is returned.
