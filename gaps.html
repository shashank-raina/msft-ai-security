Critical Gaps

🔴 Critical — Active Now

GapWhy It MattersInterim MitigationExpected Resolution
Classic Agents — no Entra security product coverage Most existing Copilot Studio agents in production are Classic Agents (Service Principals). They receive zero Entra Agent ID security product coverage: no ID Protection, no Conditional Access, no lifecycle governance. This gap is invisible from Microsoft's marketing materials. Field research confirms this is the default state of most enterprise Copilot Studio deployments. Inventory Classic vs Modern agents via AIAgentsInfo KQL; enforce end-user auth per agent via Power Platform admin; restrict org-wide sharing; manually recreate critical agents as Modern Agents Microsoft migration tool planned — no date confirmed. Manual recreation is the only current path.
Maker credentials — agent authenticates as builder, not user Copilot Studio agents authenticate to connected services as the maker (builder), not the invoking user. Combined with org-wide sharing and no authentication, a single admin-built agent extends admin permissions to every employee. This is structurally more dangerous than OBO in Copilot Studio deployments. Enforce end-user authentication per agent (Power Platform admin); restrict sharing scope via Managed Environments; PAM hygiene on developers who build agents Architectural — requires Power Platform admin enforcement. Product default is unlikely to change.
OBO — no true per-agent least privilege (non-Copilot Studio agents) Standard agents inherit invoking user's full token scope. Overprivileged users = overprivileged agents. No changes at RSAC 2026. PAM hygiene on users; Foundry Guardrails for tool whitelisting; Entra Workload Identity for app-level scoping; Defender Predictive Shielding during active attacks (preview) Dependent on Entra Agent ID GA — timeline unconfirmed

Even when it is, it will only protect Modern Agents — Classic Agent migration must happen first. Not announced as GA at RSAC 2026. Entra Workload Identity as stopgap; manual agent inventory; Agent 365 for discovery (GA May 1 2026) GA timeline not publicly committed; expected H2 2026. Migration tool needed before most orgs can benefit.

🛠️ Automate gap discovery — M365 Copilot Automated Readiness Assessment (ARA)

Before manually working through this gap register, consider running the open-source Microsoft ARA tool. Significant Gaps
GapInterim Mitigation
Identity tool fragmentation — 5 identity + 4 network tools on average
Microsoft research (March 2026) found organisations manage an average of 5 identity solutions and 4 network access solutions, often from different vendors. Nearly half of security leaders report being overwhelmed by vendor sprawl — a figure increasing year over year. In this environment, deploying Microsoft's agent security stack (Entra Agent ID, CA for Agents, Defender, Purview) alongside existing non-Microsoft identity infrastructure creates integration complexity that slows policy propagation and creates coverage gaps between tools. Classic agents are invisible to Entra products regardless of other tools deployed.		 Adopt the Microsoft Access Fabric model: consolidate identity and network access tooling where possible so policy changes propagate in near real time. At minimum, ensure Defender + Entra + Purview signals flow into a single Sentinel workspace. Audit which agents fall outside the Entra perimeter regardless of what other identity tools are in use.

🟡 Significant — Manageable with Controls


Agent 365, Security Dashboard for AI, and Entra Agent ID portal show different agent counts. Microsoft has confirmed this is a known issue. Security teams cannot rely on any single portal for a definitive agent inventory.
GapInterim Mitigation
Orphaned agents — two distinct scenarios, both undetected by default
Scenario A — Blueprint deleted (Entra): When a Blueprint is deleted, the Modern Agent Identities it created are not automatically removed. They retain all permissions (Graph API, Azure RBAC, Entra roles) but can no longer authenticate. Orphaned Agent Users remain as normal user accounts with no flag or indicator.

Scenario B — Builder left the company (most common in practice): Agents built in Copilot Studio by employees who have since left the organisation continue running with the builder's original permissions, full tool access, and no accountable owner. The agent remains active and accessible to anyone it was shared with. Microsoft does not detect or flag either scenario automatically — detection requires active KQL queries against AIAgentsInfo and cross-referencing against current employee lists.		 Run quarterly Graph API sweep to detect orphaned Agent Identities (Agent Identities whose Blueprint Principal no longer exists). Use PowerShell with AgentIdentity.Read.All scope — note Global Reader gets 403, requires Agent ID Administrator. Remove orphaned identities and revoke their permissions. See Identity page for detection scripts.
Agent sprawl — no lifecycle enforcement by default
Microsoft formally defines agent sprawl as the uncontrolled expansion of agents without adequate visibility, management, or lifecycle controls. It manifests as shadow AI (agents created without IT oversight), abandoned agents (created for temporary purposes and never decommissioned), and over-privileged agents (permissions granted and never reviewed). Agent sprawl leads to increased security risk, compliance challenges, and incident response difficulty.		 Implement Entra ID Governance lifecycle management for Modern Agents — assign sponsors, configure Access Reviews, use access packages for time-bound permissions. For Classic Agents, use AIAgentsInfo KQL to audit and track all published agents quarterly. Run Playbook 01 to detect ownerless and abandoned agents.
Microsoft Managed Policies not widely known or deployed
Microsoft Managed Policies for agents provide automatic baseline CA policies that block high-risk agents — available through Conditional Access. Many organisations are unaware these exist or have not deployed them. Without them, high-risk agent identities have no automatic blocking controls.		 Review Microsoft Managed Policies in Entra → Conditional Access → Microsoft-managed policies. Enable the agent risk baseline policy to automatically block agents flagged as high-risk by Identity Protection. Applies to Modern Agents only.
Security Copilot custom agents — maker credentials risk with elevated blast radius
Custom and partner Security Copilot agents use "Connect with existing user account" — the configuring user's credentials are stored and used for every execution regardless of who triggers the agent. Security Copilot users are typically high-privilege accounts (Security Admin, SOC engineer, Global Admin). A custom agent built by a Global Admin silently extends admin-level access to Sentinel, Defender, Entra, and threat intelligence to every user who runs it. No CA for Agents, no Identity Protection coverage. Structurally identical to Copilot Studio maker credentials but with a larger attack surface.		 Use a dedicated low-privilege service account specifically for configuring custom Security Copilot agents — not a personal admin account. Treat this account like a shared service identity: minimal permissions scoped to what the agent actually needs, monitored, no interactive use. Establish an approval gate before any custom Security Copilot agent reaches production. Audit existing custom agents and review the permissions of the accounts used to configure them.
Local AI agents (OpenClaw, GitHub Copilot CLI, Claude Code) operating outside governance
Users are installing AI agents on their devices outside MDM and security controls. These local agents can autonomously execute tasks, modify code, or access confidential information with no IT visibility. Agent 365 Shadow AI page (Frontier programme) can discover OpenClaw today; GitHub Copilot CLI and Claude Code support coming. Without Frontier programme, no native discovery mechanism exists. June 2026: Defender context mapping will show local agent blast radius including MCP servers and cloud resources reachable.		 Enrol devices in Intune and ensure Defender for Endpoint coverage. Use Agent 365 Shadow AI page (Frontier programme) to discover OpenClaw instances. Apply Intune policies to block unauthorised local agents. June 2026: Defender context mapping and runtime blocking.

⚠️ Policy caveat: Once the Block AI Agents from OpenClaw policy is enabled via Agent 365, it cannot be disabled through the Agent 365 portal. Rollback requires deleting the Intune security policy (A365 - Block OpenClaw) directly in Intune. Source: Derk van der Woude, May 2026.		 High
No pre-deployment security testing for AI agents
Most AI agents go to production without a single adversarial security test. Unlike web applications (OWASP ZAP, Snyk, DAST pipelines), AI agents have no equivalent mandatory pre-deployment security gate in the Microsoft platform. Prompt injection, system prompt leakage, excessive agency, and data exfiltration vulnerabilities (OWASP LLM01, LLM02, LLM06, LLM07) are typically only discovered after deployment. The OWASP LLM Top 10 (2025) defines 10 risk categories — most agents are untested against all ten before shipping.		 Use Microsoft PyRIT (github.com/microsoft/PyRIT) for automated pre-deployment adversarial testing. Wrap PyRIT in a config-driven scanner with OWASP mapping and CI/CD release gate (exit 0 = deploy, exit 1 = block). Run quick scans on every merge (phases 1–2), full scans pre-release (all 4 phases). See Playbook 06. Source: Microsoft Tech Community — PyRIT High
No visibility into which AI model each agent uses — EUDB compliance blind spot
Copilot Studio allows agents to use third-party models (Anthropic Sonnet/Haiku/Opus, OpenAI GPT-4o etc.). The model selection is stored in RawAgentInfo.modelNameHint — not surfaced in any admin UI or registry view. Agents using Anthropic models process data outside the EU Data Boundary regardless of tenant geo settings. No native policy to prevent makers selecting out-of-EUDB models. Source: Blue161616/Agent-Identity.		 Query AIAgentsInfo with extract(modelNameHint) to identify affected agents and their EUDB status. Consider restricting available models in Copilot Studio environment settings via Power Platform Admin Center. Run before any data residency audit. High
Any user can change another agent's authentication type
By design in Copilot Studio, any tenant user can change the authentication settings of an agent they did not create and do not own — including downgrading a published agent from authenticated to No Authentication. A single non-admin user can silently expose any published agent to anonymous access without the owner's knowledge. There is currently no platform-enforced owner-only restriction on auth type changes.		 Deploy the change-detection Sentinel Analytics Rule (see Playbooks) to alert immediately when any agent is switched to No Authentication. Audit auth type changes regularly via the AIAgentsInfo table. Restrict who can access Copilot Studio environments via Managed Environments in Power Platform Admin Center.
Cross-reference all three portals; use AIAgentsInfo Advanced Hunting table as the most granular source; treat any portal count as approximate until Microsoft resolves the consistency issue
Purview triage agent 90-day re-auth requirement
Purview Security Copilot triage agents (DLP, IRM) run in the security context of the last user who saved the config. After 90 days, the agent stops running until config is manually re-saved. No automatic renewal.		 Create a calendar reminder at 80 days post-config to re-save. Consider assigning a shared service account as the config owner to avoid disruption if the original user departs.
SharePoint oversharing — silent Copilot data exposure vector
M365 Copilot grounds responses in data the user has permission to access. Sites shared with Everyone Except External Users (EEEU), broken permission inheritance, anonymous sharing links, and ownerless sites all become Copilot exposure vectors — Copilot will surface content from these sites in responses to any user with access. Most organisations have significant oversharing debt that predates Copilot deployment. This is a data risk distinct from agent identity risks — it exists even in well-governed Classic/Modern agent environments.		 Run SharePoint Advanced Management (SAM) Content Management Assessment to identify high-risk sites. Apply Restricted Content Discovery (RCD) immediately to exclude sensitive sites from Copilot grounding while remediation is underway. Then remediate: remove EEEU access, fix broken inheritance, assign site owners, restrict sharing links. Use DSPM for AI Data Risk Assessments to continuously monitor. See Microsoft's three-step blueprint.
Real-time agent tool protection — Agent Tooling Gateway (ATG)
Agents onboarded via Agent 365's Agent Tooling Gateway (ATG) now get real-time protection equivalent to the existing Copilot Studio Defender RT integration — tool actions evaluated and blocked before execution. However, this only applies to agents explicitly onboarded via ATG. Custom agents, Classic Copilot Studio agents, and any agent not registered with Agent 365 receive no ATG protection. Coverage depends on adoption of Agent 365 SDK or ATG integration by the agent builder.		 Ensure new custom agent builds onboard via Agent 365 SDK to receive ATG real-time protection. Classic Copilot Studio agents remain outside this coverage. Treat ATG as the enforcement path for Agent 365-registered agents going forward.
DLP label-blocking previously limited to SharePoint/OneDrive — now being fixed
Until April 2026, Purview DLP label-blocking for Copilot only applied to files stored in SharePoint or OneDrive. Files on local devices, network shares, and non-Microsoft cloud locations were not covered — Copilot could process sensitivity-labelled confidential files from these locations despite active DLP policies. This was confirmed in practice by incident CW1226324 (January 2026), in which Copilot accessed confidential emails in Outlook Drafts and Sent Items for approximately one month. Resolution: rolling out mid-April to May 2026, DLP label-blocking now applies to Word, Excel, and PowerPoint files regardless of storage location. No policy changes required — existing rules apply automatically.		 ✓ Resolving April–May 2026 rollout — verify rollout has reached your tenant before treating as closed. Ensure sensitivity labels are applied to files requiring protection — unlabelled files remain unblocked regardless of DLP policies. Deploy auto-labelling policies via Purview Information Protection to cover unlabelled content at scale.
DLP coverage gap — unmanaged devices without Edge for Business
Browser-layer DLP (Edge for Business inline protection) only activates where users are signed into their Edge for Business profile. Users on unmanaged devices using Chrome, Firefox, or Safari to access external GenAI tools have no browser-layer DLP coverage. Network Data Security (Global Secure Access) fills this gap but is still in preview. Organisations have a residual exposure window for BYOD users on non-Edge browsers.		 Prioritise Edge for Business rollout as an immediate mitigation. Enable Network Data Security (Global Secure Access preview) for broader device coverage. Communicate acceptable use policy for GenAI apps on personal devices pending GA.
AI Agent Inventory — complex setup requiring two admins
Setup requires collaboration between Defender admin (enable 3 preview features) AND Power Platform admin (enable separate threat detection toggle). Takes up to 30 min for connection, longer for data population. Not self-service.		Assign a joint Defender + Power Platform admin workstream for onboarding. Verify via AIAgentsInfo KQL after setup. Plan for 30-minute minimum delay on initial data.
Agent name sync bug — Copilot Studio rename not reflected in Entra Agent ID
Agents renamed in Copilot Studio keep their original "Agent #" name in Entra. Makes per-agent CA policy management nearly impossible at scale.		Use Agent ID object-ID (not name) as primary key for agent identification. Cross-reference via PowerShell script against Power Platform Admin Environment URL. Monitor for Microsoft fix — no timeline confirmed.

Foundry Guardrails in preview, Foundry agents only — no equivalent control for Copilot Studio agentsPower Platform admin controls for Copilot Studio agents (authentication enforcement, sharing limits); Defender for Cloud Apps for API-layer controls
CopilotActivity table prompt sensitivity + ingestion cost
The Copilot Data Connector for Sentinel ingests prompt content into the CopilotActivity table — making user prompts a sensitive artifact inside the SIEM. Standard ingestion costs apply (pay-per-GB). Organisations must apply field-level masking, retention policies, and access controls on the table before enabling in production.		Enable in a controlled test workspace first. Apply field-level masking on prompt content fields. Set short retention for sensitive fields. Restrict CopilotActivity table access to SOC personnel only. Plan and budget for ingestion volume before enabling at scale.
Defender for Cloud Apps RT protection — 1-second timeout
If the Defender system doesn't return a block decision within 1 second, the agent proceeds to execute the tool anyway. Fast tool calls may bypass protection.		Ensure network latency between Copilot Studio environment and Defender is minimised. Treat as a detection tool, not a guaranteed prevention control.
ZT Assessment AI pillar not until summer 2026Use existing ZT Workshop assessment for Identity/Data/Network pillars; manually assess against ZT for AI reference architecture (published March 2026)
No built-in platform kill switch — open-source option now available
Microsoft has no single-click kill switch for all agent types. The Agent Governance Toolkit (open-source, MIT, April 2026) provides an automated kill switch with ring isolation and trust decay via Agent Runtime package. For platform controls: Entra CA for Modern Agents, Power Platform admin can disable Classic agents, Defender Predictive Shielding (preview).		Adopt Agent Governance Toolkit for programmatic kill switch capability. Pre-plan incident response runbook for agent compromise scenarios. Entra CA for Modern Agents provides policy-based blocking. Power Platform admin can disable Classic agents manually. See Playbook 04 for incident response.

KQL Detection Queries

🔍 Key Advanced Hunting Queries (AIAgentsInfo table)

These queries require the AI Agent Inventory to be enabled in Defender for Cloud Apps (requires Defender admin + Power Platform admin collaboration).

No-Auth Agents
Finds agents accessible without user authentication — highest blast radius risk combined with maker credentials.
AIAgentsInfo
| summarize arg_max(Timestamp, *) by AIAgentId
| where UserAuthenticationType == "None"
| project AgentCreationTime, AIAgentName,
  AIAgentId, AgentStatus, CreatorAccountUpn

Ownerless Published Agents
Finds published agents with no accountable owner — governance blind spots that can persist indefinitely.
AIAgentsInfo
| where AgentStatus == "Published"
| where isempty(OwnerAccountUpns)
| project AgentCreationTime, AIAgentName,
  AIAgentId, AgentStatus, CreatorAccountUpn

A well-architected deployment combining these controls with Modern Agent migration provides meaningful defence in depth — but the Classic Agent estate must be addressed first.
