Microsoft Entra Agent ID — Concept
What it is, why it matters, and how Classic vs Modern agents differ
🚨 The problem today: Most Copilot Studio agents in production are "Classic" — registered as ordinary service principals with zero agent-specific security controls. No CA for Agents, no ID Protection, no lifecycle governance. Entra Agent ID is the fix — but it's Preview, requires Frontier programme access, and only covers agents created after enabling it.
💡 The plain-English version: Today, most agents share the same badge as the person who built them. Entra Agent ID gives each agent its own badge — with its own permissions, its own risk detection, and its own expiry date. The Security Dashboard only gets rich data for agents with their own badge (Modern agents).
The four identity objects every security architect needs to know
📋
Agent Identity Blueprint
Template + Credential Container
Defines how agents are created. Holds credentials (best practice: Federated Identity Credentials — no stored secret). If deleted: credentials gone but all downstream permissions remain — orphaned permission assignments with no auth path.
⚠ Protect credentials here
🔑
Blueprint Principal
Audit Identity
Service principal created when blueprint is added to tenant. One job: create/deprovision agent identities. All lifecycle events logged under this identity in Entra audit logs. Deleted = audit trail severed.
⚠ Deletion breaks audit trail
🤖
Agent Identity
The Agent's Service Principal
The agent's own Entra identity. No credentials of its own — uses Blueprint credentials. Full Entra security stack applies: CA for Agents, ID Protection, lifecycle governance. Has Owner + Sponsor.
✓ Full Entra security applies
👤
Agent User
Optional — Full Human Identity
Some agents get a full user account (mailbox, Teams, SharePoint). Appears as a normal user in Entra — no indicator it's an agent. If Blueprint deleted: Agent User remains with all permissions, no flag, no expiry.
🚨 Survives Blueprint deletion

Setting Up Copilot Monitoring
Dual-admin setup — Defender admin + Power Platform admin must work together
⚠️ Two admins required. Neither can complete this alone. Coordinate before starting — the Power Platform admin needs a URL that only the Defender admin can generate.
1
Enable Security for AI agents in Defender
🔵 Defender Admin
Defender portal → Settings → Security for AI agents → Enable toggle
Turns on the Security for AI feature. This creates the integration point Copilot Studio registers with. Note: previously at Settings → Cloud Apps → AI Agents — navigation moved April 2026, rolling out to tenants.
2
Connect Microsoft 365 as a Cloud App
🔵 Defender Admin
Settings → Security for AI agents → Connect Microsoft 365
Enables Copilot agent telemetry to flow into Defender. Without this you get inventory but no alerts or behavioural data.
3
Copy the webhook URL from Defender
🔵 Defender Admin
Security for AI agents → Copilot Studio AI Agents → Copy URL
Defender generates a unique tenant-specific URL. Pass this to the Power Platform admin — they need it for Step 4.
4
Enable Defender integration in Power Platform
🟣 Power Platform Admin
admin.powerplatform.com → Security → Threat Detection → Microsoft Defender → Enable
Paste the webhook URL from Step 3. This instructs Copilot Studio to send agent telemetry to Defender. Enable and confirm.
⚠️ This step requires Power Platform Admin role — Defender Admin cannot do it.
5
Enable Real-Time Protection
🔵 Defender Admin
Settings → Security for AI agents → Real-time protection → Enable
Enables Defender to evaluate and block tool invocations before execution. Without this, you get inventory and alerts but no active blocking.
⚠️ 1-second timeout — Defender evaluates tool calls before execution. Fast tool calls may execute before a block decision is returned.
